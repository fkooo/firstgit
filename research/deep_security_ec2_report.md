# AWS EC2環境におけるTrend Micro Deep Security 導入検証報告書

## 1. はじめに
本資料は、AWS EC2環境に対するサーバー保護ソリューション「Trend Micro Deep Security (Cloud One - Workload Securityを含む)」の導入によって実現されるセキュリティ強化の全体像、および環境ごとの最適な運用設計の方針を報告するものです。

---

## 2. Deep Security 導入の最大のメリットと強み

他社製品や従来の運用スタイルと比較した際、Deep Securityには以下の決定的な強みがあります。これらは単なる機能の提供にとどまらず、**「運用負荷を激減させながら防御力を最大化し、クラウドの俊敏性を妨げない」**という大きなビジネス価値を生み出します。

### ① パッチ当てのダウンタイムを排除する「仮想パッチ」
脆弱性が発表された際、本番サーバーを停止して正規パッチを適用する作業は大きな負担となります。「仮想パッチ」機能は、エージェントがネットワーク層で脆弱性を突く攻撃通信のみを瞬時にブロックするため、**サーバーを無停止のままゼロデイ攻撃や修正前の脆弱性から保護**します。レガシーOSの延命にも極めて有効です。

### ② 職人技を不要にする「推奨設定の自動適用」
数万あるIPSルールを手動で選出する必要はありません。Deep SecurityはサーバーのOSや稼働中のアプリケーションを自動解析し、**「その環境に今本当に必要な仮想パッチ」だけを自動的にピンポイントで適用**します。管理者のチューニングにかかる手間を完全にゼロにします。

### ③ 「1つのエージェント」で完結する網羅的な多層防御
アンチウイルスソフト、IPS装置などを複数導入することなく、超軽量な**1つのエージェント**をインストールするだけで、「ネットワーク層」から「ファイル層（アンチマルウェアや変更監視）」までの統合的なセキュリティ保護を実現します。

### ④ AWS Auto Scaling への完全追従（クラウドネイティブ）
EC2インスタンスが負荷に応じて自動増減（オートスケーリング）する際、起動と同時にエージェントが連携し、人の手を介さずに全自動でセキュリティポリシーが適用されます。クラウド特有のスピード感を一切損ないません。

---

## 3. コア機能（多層防御）の詳細レイヤー

Deep Securityが提供する強固な防御網は、異なるレイヤーで働く以下の機能の組み合わせにより実現されます。

| 防御レイヤー | 機能名 | 保護対象・役割 | 特徴 |
| :--- | :--- | :--- | :--- |
| **ネットワーク層** | **侵入防御 (IPS/IDS)**<br>＋ **仮想パッチ** | OS・ミドルウェアの「脆弱性」を狙う攻撃 | 通信パケットを監視し、悪意ある通信がサーバーに届く「直前」でブロック。建物の入り口の警備員。 |
| **ファイル・OS層** | **不正プログラム対策**<br>(アンチマルウェア) | 不正なファイル（ウイルス、ランサムウェア等） | サーバー内部に侵入・保存されたファイルや挙動を監視し駆除する。建物の中の警察官。 |

これに加えて、「Webレピュテーション（不正サイトへの通信遮断）」や「変更監視（ファイル改ざん検知）」などを組み合わせることで、標的型攻撃の侵入から実行までの各フェーズを水際で食い止めます。

---

## 4. サーバーロール（用途）別のセキュリティポリシー設計案

システム環境（本番/開発）やサーバーの目的に応じて、セキュリティレベル（防御の強さ）を柔軟に変更・調整して適用することが推奨されます。

> [!TIP]
> Deep Securityではこれらの設定を「ポリシーのテンプレート」として事前に作成しておき、EC2のタグや用途に応じて自動割り当てを行う運用がベストプラクティスです。

> [!WARNING]
> **【よくある運用上のご質問】社内からの通信（IPアドレス）のみに制限すれば、保守サーバ等のIPS(侵入防御)はOFFにしても良いか？**
> 
> 結論から申し上げますと、**技術的にOFF（無効化）に設定することは可能ですが、セキュリティの観点（多層防御・ゼロトラスト）からは強く「非推奨（ONのままを推奨）」**となります。
> 理由は以下の通りです。
> 
> 1. **ラテラルムーブメント（脅威の横展開）の防止**: 万が一、社内従業員のPC（端末）や別の開発用サーバーが標的型攻撃やランサムウェアに感染した場合、攻撃者はそこを踏み台にして「信頼された社内ネットワーク」から保守サーバ等へ攻撃を仕掛けてきます。
> 2. **ネットワーク制御（IP制限）の限界**: EC2のセキュリティグループ等で「社内IPからのアクセス」を許可した場合、通信の中身（攻撃コードが含まれていないか等）まではチェックできません。感染した社内PCから許可されたポート経由で脆弱性を突かれた場合、エージェント側のIPS機能によるブロックが最後の砦となります。
> 3. **妥協案**: もし社内システム同士の通信において、誤検知による業務停止を強く懸念される場合は、OFFにするのではなく**「検知（IDS/アラートのみ）モード」に変更し、通信を通しつつ監視を行える状態**にしておくことを推奨いたします。

## 5. 運用ノウハウ：ポリシーのポータビリティ

Deep Security Managerで作成した上記のようなポリシー群は、**XMLファイルとしてエクスポート／インポートすることが可能**です。

*   **ユースケース**: 開発・検証用のDeep Security Managerで作成・テストを終えたポリシー（除外設定などを含む）をXMLとして書き出し、本番環境のマネージャーへそのままインポートする運用が可能です（※カスタムルールなどの一部例外を除く）。
*   注意: エクスポート元とインポート先のマネージャーバージョンは同一であることが推奨されます。

## 6. 【補足】Deep Security導入完了までの暫定保護（AWSマネージドサービスの活用）

Deep Securityの本番導入およびポリシー調整が完了するまでの間、無防備な状態を避けるため、AWSが標準で提供するマネージドサービスを活用して一定水準のセキュリティを確保することを推奨します。なお、これらはDeep Security導入後も「多層防御の外壁側」としてそのまま併用することが可能です。

### ① Web層の防御（IPS/仮想パッチの暫定代替）
*   **AWS WAF (Web Application Firewall)**
    *   **対象**: 本番Webサーバ / 開発Webサーバ
    *   **概要**: ロードバランサ（ALB等）にアタッチすることで機能します。AWS提供の「マネージドルール」を有効化するだけで、SQLインジェクション等の一般的なWeb脆弱性を突く攻撃をブロックでき、仮想パッチ導入前の水際対策として非常に有効です。

### ② 脅威の自動検知（アンチマルウェア・IDSの代替）
*   **Amazon GuardDuty**
    *   **対象**: 全てのEC2インスタンス（アカウント全体）
    *   **概要**: EC2内部に侵入したマルウェアが外部の攻撃者サーバーと通信を行う挙動などを、ネットワークログから機械学習で検知しアラートを上げます。エージェントのインストール不要で数クリックで有効化できるため、直ちに監視を開始できます。

### ③ ネットワーク層の厳格なアクセス制御
*   **EC2 セキュリティグループ**
    *   **対象**: 踏み台サーバ / 管理サーバ / 保守サーバ
    *   **概要**: 内部ネットワークであっても、SSH (22) や RDP (3389) などの管理用ポートへのアクセスを「特定の踏み台IP」や「会社の固定IP」からのみに極限まで絞り込みます。攻撃の入り口（アタックサーフェス）を物理的に狭めることで、パッチ適用前サーバへのリスクを減らします。

### ④ OSレベルの実パッチ運用
*   **AWS Systems Manager (SSM) Patch Manager**
    *   **概要**: 仮想パッチが使えない期間は、OS自体のセキュリティパッチを迅速に当てるしかありません。SSMを利用して、重要なセキュリティアップデートがリリースされた際に、複数台のEC2に対して一括で自動パッチ適用を行う仕組みを整えておきます。

### ⑤ 暫定期間中に「保護できない」領域・残存リスク（DS導入までカバーできない点）
AWSマネージドサービスによる暫定防御は強力ですが、**OS内部の深いレイヤーへのアクセス制御には限界があり、Deep Securityが導入されるまで以下の脅威に対しては現状無防備であることを認識（リスク受容）しておく必要**があります。

1.  **侵入したマルウェアの「隔離・即時駆除」**: サーバの内部（ディスク上）に直接配置されたウイルスやランサムウェアのプログラムを、実行前に駆除することはできません。GuardDutyはマルウェアが”悪意ある怪しい通信”を開始して初めて検知を行うため事後検知となります。
2.  **Web通信以外（DB・OS内部）の脆弱性保護**: AWS WAFが保護できるのは「外からのWeb通信（HTTP/HTTPS）」のみです。そのため、内部ネットワーク経由でのデータベースへの攻撃や、独自プロトコルを狙った特定のゼロデイ攻撃を止めるすべはありません。
3.  **ゼロデイ攻撃およびレガシーOSへの保護**: SSMパッチマネージャーは「メーカーがパッチを作成・公開した後」にそれを適用する仕組みです。したがって、パッチ公開前の未知の脆弱性を突く**「ゼロデイ攻撃」に対しては完全に無防備**となります。また、サポートが終了しパッチが二度と提供されないOS（CentOS 7等）への防衛手段もありません（DSの仮想パッチであればパッチ公開前・提供終了後でも即座に防御可能です）。
4.  **OS内部の「ファイル改ざん」の検知**: システムの重要ファイルやWebコンテンツがサイレントに書き換えられたことをOS内部から即座に検知する機能（変更監視機能）がないため、侵入や改ざんの発見が遅れるリスクが残ります。

## 7. クラウドネイティブ要素を取り入れたコスト最適化（削減）案

システムのセキュリティ水準を維持、あるいはさらに高めつつ、AWSならではの仕組みを活用することで、インフラ費用とDeep Securityのライセンス運用費用を削減するアーキテクチャのご提案です。

### ① 踏み台サーバ（Windows EC2）の完全廃止
ご検討の通り、旧来ネットワーク型の「踏み台サーバ」を廃止することは、コストとセキュリティ対策の両面で最も効果的なアプローチです。
*   **代替手段**: **AWS Systems Manager (SSM) Session Manager**、または **EC2 Instance Connect (EIC) Endpoint**を利用したセキュアアクセスへ移行します。
    *   **【SSM Session Managerの仕組み】**: EC2内部に常駐する「SSM Agent」が、AWS側の管理サーバ（SSMエンドポイント）に対して**EC2の内側から外向き（アウトバウンド）にHTTPS通信を確立**します。管理者がブラウザやCLIからAWSにログインすると、この張られたトンネルを通ってターミナル操作を行います。入り口（インバウンド）のポートを開けて待ち受ける必要が一切ない画期的な仕組みです。
    *   **【EIC Endpointの仕組み】**: VPCのプライベートサブネット内に、AWSが完全に管理するセキュアな「接続口（エンドポイント）」を配置します。管理者はIAMで認証されたAWS API経由でこのエンドポイントに接続し、そこから先は一時的な鍵を使って背後のEC2へ安全にルーティングされます。仮想的な踏み台サーバの運用保守をAWSに丸投げするイメージです。
    *   💻 **【対象OS（Windows / RHEL 両対応）】**: 両サービスとも、**Windows Server（RDP接続 / PowerShell）と、RHEL等のLinux（SSH接続 / CLI）の両方に完全対応**しています。ローカルPCからポートフォワーディングという機能を使うことで、これまで通りリモートデスクトップ画面やWinSCP等のクライアントソフトを使った操作も問題なく行えます。
    *   ⚙️ **【各種OSへの導入に必要な事前準備】**: 旧来の運用から切り替える際、以下の軽微な設定準備が必要となります。
        *   **(SSMの場合)**: EC2上で「SSM Agent」が起動していること（Windows Serverの公式AMIには**すべて標準インストール・起動済**です。RHEL等の最新AMIも大半は導入済です）、およびEC2に対して通信許可用の「IAMロール（AmazonSSMManagedInstanceCore）」を割り当てるだけで完了します。
        *   **(EIC Endpointの場合)**: 事前にVPC内に専用エンドポイント(ENI)を作成する作業と、対象EC2のセキュリティグループにて「そのエンドポイントからの通信のみ許可する（WindowsならRDP: 3389、RHELならSSH: 22）」というルールの追加変更が必要です。
    *   💰 **【利用コストについて（いずれも無料）】**: これらの機能自体を利用するための**「追加のAWS利用料金（時間単価等）」は一切発生しません（無料）**。※ただし、SSM Session Managerでターミナルの操作履歴（何が入力されたかの全ログ）を証跡としてAmazon S3等に保存する場合のみ、保存容量に応じたごくわずかなストレージ料金（月に数十円〜数百円程度）が発生します。
*   **コスト削減効果**: 踏み台サーバ用に24時間稼働していたEC2費用（Windowsライセンス込み）が丸ごと「無料のサービス」に置き換わるだけでなく、踏み台サーバを保護するために購入していた**Deep Securityライセンス1台分も不要**となります。
*   **セキュリティ向上**: 各サーバのインバウンドポート（SSH:22 や RDP:3389）をVPC内部からであっても完全に「閉じた状態」のまま管理者アクセスが可能になるため、ネットワークレイヤーにおける攻撃対象領域（アタックサーフェス）が劇的に減少します。

### ② Deep Security管理サーバのSaaS化（Cloud Oneへの移行）
Deep Securityのエージェントを統括する「管理サーバ（DSM）」をご自身のAWSアカウント内のEC2（Windows）として自前で構築せず、トレンドマイクロ社が提供するSaaS版である「**Trend Micro Cloud One - Workload Security**」を採用する構成です。
*   **コスト削減効果**: 管理サーバを動かすための「高いスペックを要するWindows EC2インスタンス料金」と、それに必要となる「データベース（SQL Server / RDS等）の稼働費用」がすべて丸ごと削減されます。
*   **運用負荷の激減**: 管理サーバ自体の深刻な障害対応や、システムのバージョンアップ保守といった運用作業からインフラ管理者が開放されます。
